01Project Overview

ShadowShare is a privacy-first, zero-knowledge file transfer platform. Every byte of user data is encrypted inside the sender's browser using the native Web Crypto API before a single network packet leaves the device. The server is engineered to be a blind vault — it stores opaque binary blobs, never knows file names, content, passwords, or decryption keys, and is structurally incapable of recovering them.

Even a complete compromise of the server, its database, and its object storage yields no readable user data. The decryption key never traverses any network the operator controls; it is carried only by the URL fragment (the portion after #), which browsers do not transmit to servers per RFC 3986.

Version 2.0 (this document) extends the original v1.0 cryptographic foundation with thirteen production-grade product features that move ShadowShare from a competent technical proof-of-concept to a finished, defensible privacy product fit for daily use.

Mission Statement
ShadowShare exists so that two people can transfer a file across the open internet without trusting a third party — including the operators of ShadowShare itself.

02What's New in v2.0

Version 2.0 introduces thirteen significant feature additions and a complete UX redesign. These features were selected for three reasons: each (a) closes a real-world threat surface, (b) preserves the zero-knowledge invariant, and (c) materially differentiates ShadowShare from the lineup of "encrypted file share" tools that lack mature sender control.

F1 · Burn-After-Reading NEW
Sender marks a file as single-use. Server deletes the blob atomically on first successful download.
F2 · Custom TTL NEW
Choice of 1h / 24h / 7d / 30d expiry, with a live countdown timer on the viewer page.
F3 · Sender Kill Switch NEW
Private delete-token URL returned at upload time lets the sender revoke a file at any moment.
F4 · Download-Count Status NEW
Sender-only status page (gated by delete-token) shows download count, first access, time left.
F5 · Access Password NEW
Optional second-factor password verified client-side via authenticated decryption of a token.
F6 · Link-Only Mode NEW
"Link is the key" — generate a strong random key, embed in fragment, skip the password prompt.
F7 · Multi-File / Folder Upload NEW
Client-side ZIP bundling via JSZip before encryption. Server still sees one opaque blob.
F8 · Client-Side Preview NEW
In-memory image / PDF preview from decrypted bytes — nothing is written to disk.
F9 · QR Code Share NEW
Inline QR rendering for the full share URL (including fragment) for mobile-to-mobile handoff.
F10 · Chunked Resumable Upload NEW
5 MB chunks, indexed, server-side reassembly. Recovery from connection loss without restart.
F11 · "How It Works" Explainer NEW
Interactive animated modal showing the crypto flow for non-technical recipients.
F12 · Theme System NEW
Dark / light / system preference with prefers-color-scheme support and localStorage persistence.
F13 · Security Audit Artefact NEW
SECURITY.md, entropy analysis, OWASP checklist, SSL Labs report — shipped with the repository.
Zero-Knowledge Invariant — preserved across every new feature
Every feature above was reviewed against the rule: the server must remain unable to recover any plaintext, key, password, or filename. Where a feature requires the sender to retain capability over a file (kill switch, status), that capability is rooted in a server-generated token returned only to the sender, never reconstructable by anyone else — including server operators.

03Core Principles & Threat Model

3.1 Core Principles

  1. Zero-Knowledge — server cannot decrypt any user file under any circumstance.
  2. Client-Side Encryption — AES-256-GCM via the Web Crypto API, before upload.
  3. Key Sharing via URL Fragment — the # portion of the URL is never transmitted to the server.
  4. No Authentication — possession of the link (and optionally a password) is the only credential.
  5. Ephemeral Storage — every file has an upper-bound TTL; burn-after-reading is the minimum.
  6. Sender Sovereignty — the sender retains a delete-token and can revoke at any time.
  7. Open Crypto Standards — AES-256-GCM (NIST SP 800-38D), PBKDF2 (NIST SP 800-132), HKDF (RFC 5869), SHA-256.
  8. Auditable & Open Source — full source published under MIT; reproducible builds.

3.2 Threat Model

3.3 Explicit Non-Goals

  • Identity, accounts, social graph. ShadowShare deliberately has no user accounts. Anonymity is a feature.
  • Persistent storage. Maximum TTL is 30 days. Not a backup product.
  • Defense against an attacker with physical access to a logged-in browser. Out of scope.
  • Forward secrecy across multiple recipients. A single share link is a single recipient channel; resharing the link reshares the key.

06Feature Specifications (13 New Features)

F1 — Burn-After-Reading (Single-Download Mode)

Goal: Eliminate persistence after first successful download.

UX: Sender toggles "Delete after first download" in advanced options. Viewer page displays a prominent amber warning: "This file will be permanently deleted after you download it."

Backend behaviour: When burn_after_read=true and the first GET /api/file/{id} succeeds (HTTP 200 + last byte streamed), the worker atomically deletes the row from PostgreSQL and the object from Spaces inside a transaction with SELECT … FOR UPDATE. Subsequent requests return HTTP 410 Gone.

Edge cases: Network failure mid-stream does not trigger deletion; only a complete 200 response with the final TCP byte acknowledged triggers the delete hook. Implemented via Flask's response.call_on_close().

F2 — Custom TTL with Countdown Timer

Goal: Sender controls intentional ephemerality.

UX: Segmented control with options 1h / 24h / 7d / 30d. Default 7d. Viewer page displays a live countdown ("Expires in 4h 22m") updated client-side every second.

Backend behaviour: expires_at = NOW() + interval stored at upload. Hourly cleanup job (APScheduler) deletes WHERE expires_at < NOW() from both database and Spaces.

F3 — Sender Kill Switch

Goal: Restore sender control after the link is sent.

Mechanism: At upload time the server generates a separate random 256-bit delete_token. Only its SHA-256 hash is persisted (delete_token_hash). The plaintext token is returned to the sender exactly once in the upload response and is never reconstructable.

API: DELETE /api/file/{file_id}?token={delete_token}. Server hashes the supplied token and compares to delete_token_hash; on match, deletes immediately.

UX: Success page displays two artefacts: (1) the share link, (2) a separate "Save your delete link" box clearly labelled "Only you should hold this." Copy buttons for each.

F4 — Download-Count Visibility for Sender

Goal: Sender can tell whether the recipient has opened the file — without compromising recipient privacy.

Mechanism: The same delete_token grants read access to a status endpoint: GET /api/status/{file_id}?token={delete_token}. Returns { created_at, expires_at, download_count, first_accessed_at, burn_after_read }. No IP, no user-agent, no geolocation is stored or returned.

UX: Status page at /status/{file_id}#{delete_token} with auto-refresh every 30 seconds.

F5 — Access Password (Second Authentication Layer)

Goal: Even if the share link leaks, the file remains protected.

Mechanism (client-side only):

  1. Sender enters an access password.
  2. Client derives K_access from access password via PBKDF2 (separate salt).
  3. Client encrypts a known 32-byte verifier token using K_access; uploads the resulting access_verifier with the blob metadata.
  4. Recipient is prompted for the access password; the client derives K_access locally and attempts to decrypt the verifier. AEAD authentication failure → "Incorrect password" — server never sees the password attempt.

Server role: The server stores access_verifier (opaque ciphertext) and a flag has_access_password. It does not verify the password and is structurally incapable of doing so.

F6 — Link-Only Mode ("Link is the Key")

Goal: Frictionless flow when the sender judges the channel sufficient.

Mechanism: Sender selects "Link Only" in the security mode selector. Client generates a random 256-bit AES key via crypto.getRandomValues, encrypts the file with it, embeds the raw key (base64url) in the URL fragment. Recipient opens the link, the page reads the key from location.hash, decrypts, done — no password prompt at all.

UX clarity: The mode selector explicitly labels the trade-off: "Link is the key (easier)" vs "Link + password (more secure)".

F7 — Multi-File / Folder Upload

Goal: Handle real-world transfers (multiple files, a folder of photos, a project export).

Mechanism: The drop zone accepts multiple files and folders (webkitdirectory). The client bundles them into a single ZIP via JSZip in memory, then runs the standard encrypt-and-upload pipeline. The server still sees one opaque blob and has no idea whether the contents are one file or one hundred.

UX: Selected files shown as a small list with per-file remove icons; total size shown as the chip.

F8 — Client-Side File Preview

Goal: Recipient can sanity-check before downloading.

Mechanism: After successful decryption, if the decrypted MIME type is image/* or application/pdf, the bytes are rendered:

  • Images: URL.createObjectURL(blob) into an <img> inside a sandboxed container (no script can run on a passive image element).
  • PDF: PDF.js renders page 1 only into a <canvas> from the in-memory Uint8Array.

Caveat surfaced to the user: "Preview generated from decrypted data — nothing was saved to disk."

F9 — QR Code Share

Goal: One-tap handoff to a phone in the same room.

Mechanism: Pure-client QR via qrcode npm package. Encodes the entire share URL including the # fragment. Rendered into a 200×200 canvas with brand-coloured modules. Toggleable popover next to the copy button.

F10 — Chunked Resumable Upload

Goal: Reliability on bad mobile / hotel-Wi-Fi connections.

Mechanism:

  1. POST /api/upload/init with {total_size, total_chunks, settings} returns upload_session_id.
  2. Client uploads each 5 MB chunk via PUT /api/upload/chunk/{session_id}/{chunk_index}.
  3. GET /api/upload/status/{session_id} reports last good chunk; client resumes from chunk_index + 1 on reconnect.
  4. POST /api/upload/complete/{session_id} reassembles, persists to Spaces, returns final file_id + delete_token.
Chunks are stored in a temporary scratch area inside Spaces and garbage-collected after 24 hours of inactivity.

F11 — "How It Works" Interactive Explainer

Goal: Earn the trust of non-technical recipients.

UX: Modal triggered from the header link. Three vertically-stacked animated stages: (1) browser encrypts file, (2) server stores cipher noise, (3) key travels in the URL fragment. Each stage has a small technical footnote in monospace ("PBKDF2 · 100,000 iterations · SHA-256") for the technical audience.

F12 — Theme System

Goal: Respect user preference; look correct on any device.

Mechanism: Tailwind dark: classes driven by a data-theme attribute on <html>. Initial value read from localStorage.theme; falls back to matchMedia('(prefers-color-scheme: dark)'). Toggle cycles system → dark → light → system.

F13 — Security Audit Artefact

Goal: Make the security claim falsifiable and reviewable.

Deliverable: A SECURITY.md file shipped with the repo containing:

  • Threat model (see §3.2).
  • Cryptographic design decisions with primitive choices and parameters.
  • Entropy measurement: sampled 10,000 keys/IVs, ran NIST SP 800-22 statistical test suite, report results.
  • OWASP ASVS L2 checklist with pass/fail per control.
  • SSL Labs report (target A+) screenshot and re-test link.
  • "Zero-Knowledge compliance" table mapping each invariant to evidence (code paths, test cases).

07Cryptographic Design

7.1 Primitive Choices

PurposeAlgorithmParametersStandard
Symmetric encryptionAES-256-GCM (AEAD)256-bit key, 96-bit IV (random per file), 128-bit auth tagNIST SP 800-38D
Password-based key derivationPBKDF2-HMAC-SHA-256100,000 iterations, 16-byte random salt, 32-byte outputNIST SP 800-132
Random byte generationcrypto.getRandomValues()CSPRNG provided by the browser / OSW3C Web Crypto
Hashing (delete token storage)SHA-256—FIPS 180-4
TLSTLS 1.3 (1.2 fallback disabled)Modern cipher suites only; HSTS preloadRFC 8446

7.2 Share URL Format

Two share URL formats exist, one per security mode:

// F5 — Link + Password
https://shadowshare.io/v/{file_id}#{base64url(salt)}.{flags}

// F6 — Link Only (link is the key)
https://shadowshare.io/v/{file_id}#k.{base64url(rawKey)}.{flags}

The flags segment (1–2 base64url characters) signals to the viewer whether burn-after-read or access-password modes are active, so the UI can render the correct prompts before a network round-trip.

7.3 End-to-End Crypto Flow (Upload, F5 mode)

1. user selects file F, types password P 2. salt ← crypto.getRandomValues(16) 3. key ← PBKDF2(P, salt, 100000, SHA-256, 32 bytes) 4. iv ← crypto.getRandomValues(12) 5. meta ← {filename, mime, size} as JSON bytes 6. ct_m ← AES-256-GCM(key, iv, meta) // encrypted metadata 7. ct_f ← AES-256-GCM(key, iv', F) // encrypted file body, fresh iv' 8. if access_password: K_a ← PBKDF2(P_access, salt2); verifier ← AES-GCM(K_a, iv_v, TOKEN) 9. upload { ct_f, ct_m, iv, iv', salt2?, verifier? } to server 10. server returns { file_id, delete_token, expires_at } 11. browser renders: https://.../v/{file_id}#base64url(salt).{flags}

7.4 Implementation Rules (non-negotiable)

  • Never re-use an IV with the same key. Each new ciphertext gets a fresh 96-bit IV from crypto.getRandomValues.
  • Always verify the GCM auth tag before exposing decrypted bytes to the UI. Web Crypto does this automatically; do not bypass.
  • Never serialise the key or password to localStorage, IndexedDB, or cookies — only to URL fragment / in-memory state.
  • Never include the URL fragment in any analytics, logging, error tracking, or service-worker fetch interceptor.
  • No third-party crypto libraries. Web Crypto API only.

09Database Schema

CREATE EXTENSION IF NOT EXISTS "pgcrypto";

CREATE TABLE files (
    file_id              UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    blob_path            VARCHAR(500) NOT NULL,            -- Spaces object key
    blob_size_bytes      BIGINT       NOT NULL,
    created_at           TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    expires_at           TIMESTAMPTZ  NOT NULL,
    download_count       INTEGER      NOT NULL DEFAULT 0,
    first_accessed_at    TIMESTAMPTZ  NULL,
    burn_after_read      BOOLEAN      NOT NULL DEFAULT FALSE,
    has_access_password  BOOLEAN      NOT NULL DEFAULT FALSE,
    access_verifier      BYTEA        NULL,                -- opaque ciphertext (F5)
    encrypted_metadata   BYTEA        NOT NULL,            -- opaque ciphertext
    encryption_iv        BYTEA        NOT NULL,            -- 12 bytes
    delete_token_hash    BYTEA        NOT NULL,            -- SHA-256(delete_token)
    CHECK (blob_size_bytes > 0 AND blob_size_bytes <= 104857600)
);

CREATE INDEX idx_files_expires_at ON files (expires_at);

CREATE TABLE upload_sessions (
    session_id           UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    total_size_bytes     BIGINT       NOT NULL,
    total_chunks         INTEGER      NOT NULL,
    received_chunks      INTEGER[]    NOT NULL DEFAULT '{}',
    settings             JSONB        NOT NULL,
    created_at           TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    expires_at           TIMESTAMPTZ  NOT NULL DEFAULT NOW() + INTERVAL '24 hours'
);

CREATE INDEX idx_upload_sessions_expires ON upload_sessions (expires_at);
What is intentionally absent
No columns for: filename, content-type, sender IP, recipient IP, password, password hash, encryption key, salt, user-agent, geolocation, referer. Their absence is the zero-knowledge guarantee.

10User Flows

10.1 Sender Flow (happy path)

  1. Open shadowshare.io.
  2. Drag & drop one or more files (or a folder).
  3. Choose security mode: Link+Password or Link-Only.
  4. (Optional) Open advanced options: TTL, burn-after-read, max downloads.
  5. Click Encrypt & Upload.
  6. See progress phases: deriving key → encrypting → uploading.
  7. Receive: share link + QR + delete link.
  8. Copy share link; send via preferred channel.
  9. Optionally bookmark the delete link / status page.

10.2 Recipient Flow (happy path)

  1. Open the share link.
  2. (If F5 mode) Enter access password → client-side verification.
  3. (If F5 mode) Enter encryption password.
  4. Watch decryption progress (auth-tag verification).
  5. See preview if applicable (image / PDF page 1).
  6. Click Download.
  7. (If burn-after-read) See the file vanish — server returns HTTP 410 on retry.

10.3 Sender Revocation Flow (F3)

  1. Open the delete link.
  2. Confirm intent.
  3. Backend deletes blob + row; status flips to "Deleted".

14.2 Critical E2E Scenarios

  1. Upload single file → download in second tab → bytes match.
  2. Upload with access password → wrong password → "Incorrect password" → right password → bytes match.
  3. Burn-after-read → first download succeeds → second download returns 410.
  4. Custom TTL (1h) → file expires → 410 after sweep.
  5. Kill switch → delete → 410 on share link.
  6. Status page shows download count and first-access time.
  7. Multi-file upload → recipient gets a ZIP that contains all originals byte-identical.
  8. Resumable upload survives killing chunk #3 mid-flight.
  9. Tampered blob → viewer surfaces "Integrity check failed", no download offered.
  10. Link with non-existent file_id → "Not found" state.

18Glossary

AEAD
Authenticated Encryption with Associated Data. Provides confidentiality + integrity in one primitive.
AES-256-GCM
Symmetric AEAD cipher chosen for ShadowShare's bulk encryption. NIST SP 800-38D.
Auth Tag
The 128-bit tag GCM appends; verifies that ciphertext has not been altered. Mismatch → decryption refused.
Blind Vault
A storage server engineered to be incapable of reading what it stores.
Blob
An opaque binary object. In our system, the ciphertext + IV concatenation persisted to Spaces.
Burn-After-Read
Mode in which the server destroys a blob on first successful download.
CSPRNG
Cryptographically Secure Pseudo-Random Number Generator. Provided by crypto.getRandomValues.
Delete Token
Random 256-bit secret returned to the sender once at upload; lets them revoke or read status.
IV / Nonce
Initialisation Vector. 96-bit value, unique per ciphertext, prevents AES-GCM key compromise on collision.
PBKDF2
Password-Based Key Derivation Function 2. Stretches a low-entropy password into a high-entropy key.
URL Fragment
The portion of a URL after #. Per RFC 3986 it is never sent to the server.
Web Crypto API
Native browser cryptographic interface. Used here as the sole crypto provider.
Zero-Knowledge
Property that the operator of a service cannot recover the data the service stores.
